## Context

The server has an in-memory `Index` class that provides `getWork`, `getAllWorks`, `searchWorks`, `upsert`, and `remove` — plus cross-entity methods like `getEditionsByWork` and `getCopiesByWork`. File I/O primitives (`readFile`, `writeFile`, `deleteFile`) handle atomic markdown file operations. Slugs are generated by `generateSlug`, which is the sole slug authority. The server currently has three routes: health check, config read, and config write.

This change adds the first entity CRUD API — Work — establishing patterns for all subsequent entity APIs.

## Goals / Non-Goals

**Goals:**
- Full CRUD for Work entities: create, read (list + single), update, delete
- Slug auto-generated by `generateSlug` on create, never modified on update
- Re-read file from disk before PATCH to merge external changes
- List endpoint supports search (`?q=`), sort (`?sort=`), and order (`?order=`)
- Single-work endpoint resolves edition count and copy count from the index
- Orphan-protected deletion: refuse if editions exist, cascade override via `?cascade=true`
- `created_at` set on creation, never modified
- Set the code pattern for all subsequent entity APIs

**Non-Goals:**
- Body regeneration (written as an empty placeholder for now)
- Genre normalization (a later concern)
- Author slug resolution (author links are raw wikilinks in the request)
- Frontend work pages
- Bulk operations or batch endpoints

## Decisions

### Route file organization: `server/src/routes/works.ts`

Each entity type gets its own route file under `server/src/routes/`. The route file exports a function that registers routes on an Express Router. This keeps `index.ts` lean and makes each entity's API self-contained. The Index instance is accessed via `req.app.locals.index`.

**Alternative considered**: All routes in `index.ts`. Rejected — doesn't scale past 3 entities. Separate files per entity is the standard Express pattern for APIs with multiple resource types.

### Router accepts library path as a parameter

`createWorksRouter(index, libraryPath)` takes both the Index and the library path as explicit parameters. The router does not call `readConfig()` internally. This avoids coupling the route handlers to the global config state and enables test isolation — tests pass a temp directory while production passes the real config's `library_path`.

**Alternative considered**: Reading config inside the router at module init time. Rejected — makes tests impossible to isolate without mocking the config module. Passing the path as a parameter keeps the router a pure function of its inputs.

### Re-read before write on PATCH

Before writing an updated Work to disk, the handler re-reads the current file with `readFile`, deep-merges the incoming PATCH fields into the existing frontmatter, and writes the result. This ensures fields the PATCH didn't touch (including those edited externally in Obsidian) are preserved.

**Merge rules:**
- Fields present in the request body: overwrite
- Fields absent from the request body: keep from on-disk file
- `slug`, `type`, `created_at`, `_schema`: never overwritten from request body

**Alternative considered**: Blind write without re-reading. Rejected — violates the project convention of merging external changes.

### Slug on create: generated, never user-provided

`POST /api/works` accepts `title` and optional `author` (first author name string). The slug is generated by `generateSlug(title, existingSlugs, author)` and is never exposed as an input field. This keeps slugs consistent and prevents user errors.

**Alternative considered**: Accepting an optional `slug` in the request body for user overrides. Rejected — `generateSlug` is the sole slug authority per the slug-generation capability.

### Slug collision detection: collect all existing slugs from the Index

Before calling `generateSlug`, the create handler collects every existing slug across all entity types by iterating the Index's public `getAll*` methods into a `Set<string>`. This set is passed to `generateSlug` as `existingSlugs` so the algorithm can detect collisions and apply disambiguating suffixes. The Index gained a `getAllNoteSlugs()` method to support this without exposing its private `notes` map.

**Alternative considered**: Tracking slugs in a separate registry. Rejected — the Index already holds all slugs; duplicating them in a second structure adds consistency risk.

### Sort and order on GET /api/works

The `?sort=` parameter accepts `title`, `author`, or `created_at`. The `?order=` parameter accepts `asc` or `desc` (default: `asc` for title/author, `desc` for created_at). When `?q=` is provided, search is delegated to `index.searchWorks(query)`, and then sorting is applied to the results. When no `?q=` is provided, all works are fetched and sorted.

Sorting by author sorts by the first author's name (resolved from the index).

Search matches against title, author names, genres, and aliases — all case-insensitive partial substring matches.

**Alternative considered**: Case-insensitive sorting in the database layer. Rejected — the index is in-memory, so sorting is a simple `Array.sort()` with `localeCompare`.

### Orphan protection on delete

`DELETE /api/works/:slug` checks `index.getEditionsByWork(slug)`. If editions exist, it returns `409 Conflict` with a message listing the edition count. The client can retry with `?cascade=true` to force deletion.

Cascade deletes: the work file, all linked edition files, all linked copy files (found via each edition), and all notes linking to those copies. Files are deleted via `deleteFile` and index entries via `index.remove`. The traversal order is top-down: for each edition of the work → for each copy of that edition → for each note of that copy. Leaf nodes (notes) are deleted first, then copies, then editions, then the work itself.

**Alternative considered**: Soft-delete with a `deleted: true` field. Rejected — the project uses hard deletes with confirmation. The cascade is explicit (`?cascade=true`) so the user always opts in.

### Validation: required fields only

Only `title` is required on create. All other fields are optional and default to absent (no empty arrays or nulls). On PATCH, `title` is validated as non-empty if present. Invalid fields (unknown keys) are silently ignored — this is permissive by design to support evolving the schema without breaking older clients.

**Alternative considered**: Strict schema validation with rejected unknown fields. Rejected — too rigid for a personal app where the user may add custom frontmatter fields in Obsidian.

## Risks / Trade-offs

- **Cascade delete is destructive**: `DELETE /api/works/:slug?cascade=true` removes editions, copies, and notes with no undo. [Mitigation: The frontend will show a confirmation dialog listing what will be deleted before sending the request.]
- **No request body validation beyond `title`**: Malformed data (wrong types, missing required nested fields) could create broken entity files. [Mitigation: Accept for now — the admin can fix broken files manually. Add Zod or similar validation later if it becomes a problem.]
- **Sorting by author does an index lookup per work**: O(n) author lookups for sort. [Mitigation: Acceptable for personal libraries. Can pre-index author names in a future change.]
